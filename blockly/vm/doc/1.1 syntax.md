# Syntax

DGIR programs are represented by a nested JSON object representing the program's AST. The core language does not describe any higher-level constructs such as functions, loops or conditionals; these are provided by dialects.

Just like MLIR, DGIR programs can be envisioned as a graph of nodes, called Operations, and edges, called Values between Operations. Differently to MLIR, DGIR does not follow the SSA form and instead allows for re-use of values between Operations. However, each Value's lifetime is limited to the region it is defined in, and reachability is enforced by the compiler.
Operations are contained in blocks, and blocks are contained in regions(scopes).
Operations can also define their own regions allowing for hierarchical structures such as nested functions, loops, closures, etc.

Parsing of the DGIR clear text format is handled by the dialects themselves, allowing for custom assembly formats to be defined. While there are some formal guidelines for the DGIR syntax, they are not enforced by the compiler.

Every operation can hold a source position that indicates where in the original source code this operation was generated from. This is useful for debugging and error reporting.
The syntax for the source position is a simple `loc(file, lineStart:columnStart, lineEnd:columnEnd)` expression which comes at the end of the operation.

For example:
```
%x: int32 = arith.const(42) loc(./main.java, 10:4, 10:14);
```

These are some of the main syntactical guidelines of DGIR:
- Operations:
    - **MUST** start with a prefix indicating the dialect they belong to (e.g. `arith.` for the arithmetic dialect) and a name (e.g. `const`).
        - Exception to this rule is the builtin dialect that does not require a prefix.
    - can define results which are then assigned to values (e.g. `%result: int32 = arith.const(42);`)
    - complex operation signatures which represent more than just the passing of values should be defined using a 'statement block' (e.g. `scf.for[%iv: int32 = %lb to %ub step 1]`) so that it is clear that custom semantics are involved.
    - need to be terminated with a semicolon (`;`).
- Blocks:
  - can be empty
  - start with a label followed by a colon (e.g. `.then:`).
  - can define values which can be passed during branching
    ```
    .entry:
        cf.br(.then(%foo));

    .then[%x: int32]:
        io.print(%x);
    ```
  - cannot be nested inside other blocks
  - can only be entered by a branching operation (e.g. `cf.br(.then);`). This is a hard requirement to ensure that the compiler can correctly determine the reachability of values.
- Regions:
  - belong to an operation and can contain blocks and operations. Note that operations can contain new regions allowing for hierarchical structures.
  - cannot be nested inside other regions.
  - can be empty
- Attributes:
  - Can be attached to Operations and define metadata about them, such as their type or configuration options. This could be used to define specific behavior for an operation, by, for example, defining a `mode` attribute on a comparison operation or the like.
- Values:
  - values have to start with a `%` character followed by an identifier (e.g. `%result`).
  - values get annotated with their type using a colon (e.g. `%result: int32`).
  - values are assigned using the `=` character (e.g. `%result: int32 = arith.const(42);`)
- Types:
  - Types are defined in the dialects themselves and can be as simple as primitive types such as `int32` or `bool`, or more complex types such as pointers or arrays.
  - Types can have parameters, noted by angle brackets (e.g. `mem.ptr<int32>`). The number of parameters is not enforced by the compiler and can be used to define custom types. However, the dialect defining the type should enforce the correct number and type of parameters.
  - Again, dialect specific types are prefixed with the dialect name (e.g. `mem.ptr<int32>`)
- Comments:
  - Comments start with `//` and can span multiple lines using the `/* */` syntax.

## Examples
### Factorial
```
program { // Program region start
    // Example of a main function
    func.func @main[] { // Function region start
    // Implicit entry block without arguments
        // Constant operation with value taken from 'value' attrbiute
        %const_0: int32 = arith.const(42);
        // Call to function 'fac' with argument const_0
        %result_0: int32 = func.call(@fac, %const_0);
        // Print result of operation
        io.print(%result);
    } // Function region end

    // Example of a function
    func.func @fac[%x: int32] { // Function region start
    // Implicit entry block with argument 'x'
        %const_0: int32 = arith.const(1);
        %bool_0: bool = arith.lti(%x, %const_0);
        cf.br_cond(%bool_0, .then, .else);
    // Then block
    .then:
        %const_1: int32 = arith.const(1);
        func.return(%const_1);
    // Else block
    .else:
        %const_2: int32 = arith.const(1);
        %result_0: int32 = arith.sub(%x, %const_2);
        %rec_0: int32 = func.call(@fac, %result_0);
        %result_1: int32 = arith.mul(%x, %rec_0);
        func.return(%result_1);
    } // Function region end
} // Program region end
```

### For-Loop
#### C Code
```c
int main() {
    int x = 0;
    for (int i = 0; i < 10; i++) {
        int mid = 5;
        if (i < mid) {
            x += 1;
        }
        int factor = 2;
        x *= factor;
    }
```

#### DGIR Code With SCF-Extension
```
program {
    func.func @main[] {
        %x: int32 = arith.const(0);
        %one: int32 = arith.const(1);
        %lb: int32 = arith.const(0;
        %ub: int32 = arith.const(10);
        scf.for[%iv: int32 = %lb to %ub step 1] {
            %mid: int32 = arith.const(5);
            %cond: bool = arith.lti(%i, %mid);
            cf.if(%cond) {
                %x: int32 = arith.addi(%x, %one);
            }
            %factor: int32 = arith.const(2);
            %x: int32 = arith.muli(%x, %factor);
        };
        io.print(%x);
        func.return();
    }
}
```

#### DGIR Code
```
program {
    func.func @main[] {
        %x: int32 = arith.const(0);
        %i: int32 = arith.const(0);
        %one: int32 = arith.const(1);
        cf.br(.for.header);

    .for.header:
        %max: int32 = arith.const(10);
        %cond: bool = arith.lti(%i, %max);
        cf.br_cond(%cond, .for.body, .exit);

    .for.body:
        %mid: int32 = arith.const(5);
        %mid_cond: bool = arith.lti(%i, %mid);
        cf.br_cond(%mid_cond, .then, .endif);

    .then:
        %x: int32 = arith.addi(%x, %one);
        cf.br(.endif);

    .endif:
        %factor: int32 = arith.const(2);
        %x: int32 = arith.muli(%x, %factor);
        cf.br(.for.latch);

     .for.latch:
        %i: int32 = arith.addi(%i, %one);
        cf.br(.loop);

     .exit:
        io.print(%x);
        func.return();
    }
}
```
